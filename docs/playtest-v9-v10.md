# OVERMIND 플레이테스트 프로토콜 — v9 ↔ v10 비교

> 목적: "v10이 v9보다 **더 나은 게임인가**"를 체감이 아니라 **통제된 비교**로 판정.
> 기준선 v9 = 제출 동결 커밋 `de5760d` · v10 = 현재 `HEAD`.
> 원시 데이터는 이 문서의 표(또는 종이/시트)가 **진실 원본**. 리더보드 KV는 쓰기 무보증(유실 이력)이라 보조 지표로만 쓴다.

---

## 0. "더 나음"의 정의 (측정 대상 4축)

게임이라 "재미"만 물으면 답이 안 나온다. 아래 4축으로 쪼개 각각 측정한다.

| 축 | 무엇을 보나 | 왜 중요 (OVERMIND 관점) |
|---|---|---|
| **A. 공정한 난이도** | 죽었을 때 "내 실수였다"고 납득했나 (억까 아님) | v10 설계 원칙 = "스탯 말고 지능". 텔레그래프 보존 → 회피 가능해야 공정 |
| **B. 지각되는 지능** | 보스/적이 "나를 읽고 대응한다"고 느꼈나 | **핵심 가치 축.** 이게 안 느껴지면 v10의 존재 이유가 없음 |
| **C. 재미·몰입** | 한 판 더 하고 싶었나 | 최종 사용자 만족 |
| **D. 도달 성과** | 도달 웨이브·점수·생존시간 | 보조(객관) 지표. 난이도 곡선 확인용, "재미"의 대리지표 아님 |

**사전에 정한 성공 기준 (분석 전에 못박기 — 사후 합리화 방지):**
> v10이 v9 대비 **B(지각 지능) 평균 +1.0 이상** 그리고 **A(공정성) 동률 이상**이면 "개선 성공".
> B가 안 오르면 나머지가 좋아도 "방향 실패"로 본다.

## 1. 셋업 — 두 빌드 동시 구동 (블라인드)

두 버전을 별도 포트로 동시에 띄우고, 테스터에겐 **"빌드 A / 빌드 B"** 로만 부른다.

```bash
# v9 (동결 기준선) — 격리 워크트리에 체크아웃 후 프로덕션 빌드 미리보기
git worktree add /tmp/overmind-v9 de5760d
cd /tmp/overmind-v9 && npm install && npm run build && npm run preview -- --port 5109
#   → http://localhost:5109  (게임 내 버전표기 "v9")

# v10 (현재) — 별도 터미널
cd /data/game && npm run build && npm run preview -- --port 5110
#   → http://localhost:5110  (게임 내 버전표기 "v10")
```

- **프록시는 공유**: 두 빌드 모두 같은 배포 워커(`overmind-proxy…workers.dev`)의 LLM·프롬프트를 쓴다. 즉 보스 두뇌는 상수로 고정되고 **클라이언트 게임 로직 차이만** 분리 측정된다. (원하면 정답)
- **리더보드 자동 분리**: 각 빌드가 `GAME_VERSION`으로 태깅 → v9/v10 버킷에 알아서 들어감. 단 유실 가능하므로 아래 기록지가 우선.
- **블라인드 강화(선택)**: 화면 우하단 버전표기가 "v9/v10"으로 보여 힌트가 샌다. 신경 쓰이면 두 빌드의 표기를 임시로 "A"/"B"로 덮는 1줄 패치 요청. 가족 테스터면 "어느 게 새 거"라고 말하지 않는 것만으로도 충분.

## 2. 참가자 · 판수 · 순서

현실적 소표본(가족·지인·심사자). 통계적 유의성은 목표 아님 — **방향성 신호**를 본다.

- **참가자**: 최소 3명 이상 권장. 숙련도 라벨 기록(초심자/경험자).
- **워밍업 폐기**: 각 참가자, 각 빌드에서 **첫 1판은 조작 익히기용 → 데이터 버림.**
- **본 판수**: 각 빌드 **3판씩**(워밍업 제외). 즉 1인당 (1+3)×2 = 8판.
- **순서 카운터밸런싱** (학습효과 상쇄 — 이게 제일 중요):
  - 참가자 홀수번: **A→B** 순
  - 참가자 짝수번: **B→A** 순
  - "빌드 A=v9/v10" 매핑도 참가자마다 뒤집어, A라는 글자에 붙는 선입견 제거.

| 참가자 | 순서 | A = ? | B = ? |
|---|---|---|---|
| P1 | A→B | v9 | v10 |
| P2 | B→A | v10 | v9 |
| P3 | A→B | v10 | v9 |
| … | (교대) | (교대) | (교대) |

## 3. 진행 스크립트 (참가자 1명 세션)

1. "빌드 몇 개를 비교 중이고, 어느 게 새 건지는 안 알려드립니다. 편하게 플레이하고, 죽으면 왜 죽었다고 느꼈는지만 말해주세요."
2. **빌드 1 (그 참가자의 첫 빌드)**: 워밍업 1판 → 본 3판. 각 판 종료마다 §4 기록지 채움.
3. 빌드 1 3판 끝 → **§5 설문(빌드 1)** 작성.
4. **빌드 2**: 워밍업 1판 → 본 3판. §4 기록.
5. **§5 설문(빌드 2)** 작성.
6. 마지막에 **비교 1문항**: "두 빌드 중 어느 쪽이 더 '나를 읽는다'고 느꼈나요? 왜?" (자유서술)

| 참가자 | 빌드(v9/v10) | 판# | 도달웨이브 | 점수 | 생존(초) | 사망원인(무엇에/어디서) | "억까였나?"(Y/N) | ANOMALY EMP 발동? | 메모 |
|---|---|---|---|---|---|---|---|---|---|
| | | 1 | | | | | | | |
| | | 2 | | | | | | | |
| | | 3 | | | | | | | |

- **사망원인**: "보스 예측탄 정면", "쫄 무리 포위", "방패맨 접근 실패" 등 구체적으로.
- **억까 여부**: 회피 불가능했다고 느끼면 Y → A(공정성) 축의 핵심 데이터.
- **ANOMALY EMP**: v10 신규 mindgame 발동 경험. v9엔 없음(→ v9는 N/A).

## 5. 세션 후 설문 (주관 · 빌드마다 1회, 5점 척도)

각 문항 **1(전혀 아니다) ~ 5(매우 그렇다)**.

1. **[A-공정]** 죽었을 때 대체로 내 실수라고 납득됐다.
2. **[B-지능]** 보스/적이 내 플레이 습관을 읽고 대응한다고 느꼈다.
3. **[B-지능]** 같은 패턴을 반복하면 처벌받는다는 걸 체감했다.
4. **[C-재미]** "한 판 더" 하고 싶었다.
5. **[C-몰입]** 적의 움직임/타격감이 만족스러웠다. (손맛)
6. **[자유]** 가장 짜증났던 순간 1개 / 가장 좋았던 순간 1개.

## 6. 교란요인 통제 체크리스트

- [ ] **학습효과**: 워밍업 폐기 + 순서 카운터밸런싱 적용했나? (안 하면 나중에 한 빌드가 무조건 유리)
- [ ] **순서/라벨 편향**: A/B ↔ v9/v10 매핑을 참가자마다 뒤집었나?
- [ ] **환경 동일**: 같은 기기·화면·입력장치(키보드 vs 모바일 섞지 말 것). 프레임 드랍 있으면 메모.
- [ ] **네트워크**: 프록시 응답 지연/폴백 발생 시 그 판 메모(보스가 멍청해졌을 수 있음 → 지능 점수 왜곡).
- [ ] **리더보드 유실**: 기록지(§4)를 진실 원본으로. 리더보드 숫자와 어긋나면 기록지 채택.

## 7. 분석 — 어떻게 읽나

소표본이므로 **평균 + 개별 코멘트**를 함께 본다. p값·유의성 주장 금지.

1. **축별 평균 비교표** 작성:

   | 축 | v9 평균 | v10 평균 | Δ |
   |---|---|---|---|
   | A 공정(문1) | | | |
   | B 지능(문2·3 평균) | | | |
   | C 재미(문4·5 평균) | | | |
   | D 도달웨이브(중앙값) | | | |
   | 억까율(억까 Y / 전체 사망) | | | |

2. **사전 성공 기준 대조**: B가 +1.0 이상이고 A가 동률↑ → "개선 성공". 아니면 실패 원인 진단.
3. **비교 1문항(§3-6) 정성 분석**: "어느 쪽이 나를 더 읽나"에 v10을 고른 비율과 *이유*. 이게 핵심 가치의 직접 증거.
4. **억까 사망 패턴**: v10에서 특정 사인(예: 보스 예측탄)이 억까로 몰리면 → `config.ts`의 `ai.leadFactor` 등 후속 튜닝 타깃.
